这些基本就是我们之前设计的方向：
文件资源归文件资源，TMDB 元数据归 TMDB 元数据，播放进度单独拆表。

尤其这几个点是对的：

media_files                 本地/WebDAV 文件

而且你的外键也基本建好了，例如：

playback_progress.file_id
REFERENCES media_files(id)
ON DELETE CASCADE

这说明：删除某一集的 media_files 后，对应播放进度和最近播放理论上会一起删除。

目前最主要的问题：同一个文件夹被存了 3 份

我看到 source_folders 里同一个逻辑目录出现了多种路径形式。

比如《家业》有这几条：

439  /夸克/来自：分享/家业/
440  /dav/夸克/来自：分享/家业
964  /dav/夸克/来自：分享/家业/

《低智商犯罪》也有类似重复：

1200  /本机存储/低智商犯罪/
1202  /dav/本机存储/低智商犯罪
1779  /dav/本机存储/低智商犯罪/

这会带来一个很实际的问题：
文件、匹配、文件夹偏好没有挂到同一个 folder_id 上。

比如现在《家业》：

media_files.folder_id = 440
source_folder_matches.folder_id = 440
folder_preferences.folder_id = 964

也就是说，文件和 TMDB 绑定在 folder_id = 440，但横屏偏好存在 folder_id = 964。

这样如果你取消勾选《家业》文件夹，只删除 folder_id = 440，那么：

media_files 会删
media_file_matches 会删
source_folder_matches 会删

但：

folder_preferences.folder_id = 964

可能不会删，因为它挂在另一个重复文件夹记录上。

这不符合我们之前设计的“取消勾选文件夹时，本地状态一起清理”。

建议修复

必须做路径规范化，保证同一个逻辑文件夹只生成一个 source_folders 记录。

比如统一规则：

1. 不把 /dav 当作业务路径的一部分
2. folder path 统一以 / 开头
3. folder path 统一以 / 结尾
4. source.base_url 负责保存 https://.../dav
5. source_folders.path 只保存 /夸克/来自：分享/家业/

也就是说，《家业》最终只应该保留：

/夸克/来自：分享/家业/

不要同时保存：

/dav/夸克/来自：分享/家业
/dav/夸克/来自：分享/家业/
第二个问题：app_state 里仍然保存完整旧 JSON 和明文密码

你现在虽然已经有了新版表结构，但 app_state 里仍然有：

key = media_state

里面还保存了：

"sources": [
  {
    "username": "admin",
    "password": "...",
    "selectedPaths": [...]
  }
],
"items": [...],
"progress": {...},
"durations": {...},
"lastPlayedAt": {...},
"folderOrientations": {...}

这里有两个问题。

第一，数据重复。
items 已经迁移到 media_files，progress / durations / lastPlayedAt 已经迁移到 playback_progress，folderOrientations 已经迁移到 folder_preferences，旧 JSON 继续保留会造成双写或不一致。

第二，安全问题。
app_state.json.sources 里仍然有 WebDAV 明文密码。新版 sources 表已经用了：

credential_id = source:1781168620304560

这是正确方向，但旧 JSON 里的密码应该删除或不再写入。

建议修复

迁移完成后，app_state.media_state 不应该继续作为数据源。

可以改成只保存轻量状态，例如：

{
  "version": 2,
  "migrated": true,
  "lastMigrationAt": 1781168836267
}

或者干脆不再保存 media_state 大 JSON。

第三个问题：旧表还在，而且还有数据

你现在还保留了旧表：

metadata
metadata_titles
metadata_episodes
metadata_images

其中：

metadata_titles      2 条
metadata_episodes    2 条
metadata_images      20 条

如果这是为了兼容旧版本，可以接受。
但如果新版 App 已经改用：

tmdb_tv_shows
tmdb_tv_seasons
tmdb_tv_episodes
tmdb_images
image_cache

那旧表最好只在迁移阶段使用，不要继续作为业务数据源。

否则会出现：

tmdb_tv_episodes 有 26 条
metadata_episodes 只有 2 条

这两个来源明显不是同一套完整数据，后续很容易混乱。

建议

明确规则：

新版本读取：
tmdb_tv_shows / tmdb_tv_seasons / tmdb_tv_episodes / tmdb_images / image_cache

旧表：
只用于 migration，不参与正常业务读取

迁移成功后可以选择清空旧表，或者保留但不再写入。

第四个问题：季信息不够完整

你现在的 tmdb_tv_seasons 有 2 条，但季表里的部分字段是空的：

name = null
episode_count = null

而 tmdb_tv_shows 里显示：

家业 number_of_episodes = 42
低智商犯罪 number_of_episodes = 24

但是本地 tmdb_tv_episodes 里是：

家业 24 集
低智商犯罪 2 集

这个不是错误，因为你可能只拉取了本地已有集数。
但如果你请求的是 TMDB season details，正常应该能拿到这一季的 season name、overview、air_date、poster_path、episode_count 等信息。

建议

tmdb_tv_seasons 应该至少保存：

season_number
name
overview
air_date
episode_count
poster_path

如果这些字段拿不到，也应该保留 raw_json，并且 last_synced_at 记录同步时间。

第五个问题：取消勾选文件夹时，当前重复 folder_id 会影响清理

按照你现在结构，如果取消勾选某一集：

DELETE FROM media_files WHERE id = ?;

这一集对应的：

media_file_matches
playback_progress

会通过外键一起删，这一点是符合规划的。

但如果取消勾选整个文件夹，现在要小心，因为存在重复 folder：

文件在 folder_id = 440
偏好在 folder_id = 964

所以只删一个 source_folders.id 不一定能删干净。

正确的清理应该是

先做路径规范化，找到同一逻辑目录下所有重复 folder_id：

/夸克/来自：分享/家业/
等价于：
/dav/夸克/来自：分享/家业
/dav/夸克/来自：分享/家业/

然后在事务里一起删：

BEGIN TRANSACTION;

DELETE FROM source_folders
WHERE id IN (440, 964, 439);

COMMIT;

这样才能确保：

media_files
media_file_matches
playback_progress
source_folder_matches
folder_preferences
match_tasks
match_candidates

都被级联清理。

更推荐的是：先修复数据，不要让重复 folder_id 继续出现。

我会这样评价：

表结构设计：        85% 符合

播放进度设计：      符合
最近播放设计：      符合
TMDB 元数据拆分：   符合
图片缓存设计：      符合
路径规范化：        需要修
旧 JSON 迁移清理：  需要修
旧 metadata 表：    需要明确是否废弃
最应该优先改的 3 件事

第一，统一 folder path 规范，避免 /dav/...、无 /dav/...、末尾斜杠不同导致重复文件夹。

第二，迁移完成后不要继续使用 app_state.media_state 大 JSON，尤其不要再保存 WebDAV 明文密码。

第三，让 folder_preferences、source_folder_matches、media_files 挂到同一个 source_folders.id。否则取消勾选文件夹时不能保证清干净。

总结：
这个数据库的大方向是对的，已经基本按我们规划拆出来了；但现在最大的问题是路径没有规范化，导致同一个文件夹生成多个 folder_id。这个必须修，否则文件夹级删除、偏好清理、增量扫描都会有隐患。
